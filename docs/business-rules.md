# Business Rules

## Статусы КП

```
draft ──→ sent ──→ accepted
                ↘ rejected ──→ draft
```

| Статус | Значение | Редактирование | Кто устанавливает |
|--------|----------|----------------|-------------------|
| `draft` | Черновик | Полное | manager, admin, owner |
| `sent` | Отправлен клиенту | Только статус | manager, admin, owner |
| `accepted` | Клиент принял | Только просмотр | admin, owner |
| `rejected` | Клиент отклонил | Только просмотр | admin, owner |

Допустимые переходы: `draft→sent`, `sent→accepted`, `sent→rejected`, `rejected→draft`  
Реализовано в `shared/types/Kp.ts` → `KP_STATUS_TRANSITIONS`

Ограничение редактирования на фронтенде реализовано: при `sent` и `accepted` редактор КП работает в режиме read-only.

## RBAC роли и permissions

Роли хранятся в отдельной коллекции `roles` (не фиксированы в коде):
- системные: `owner`, `admin`, `manager`, `viewer` (`isSystem = true`);
- кастомные: создаются через админку (`isSystem = false`).

Правила системных ролей:
- `owner` и `admin`: permissions read-only в админке (изменение запрещено API);
- `manager` и `viewer`: permissions редактируемы через админку.

Правила enforcement:
- backend authoritative: каждый защищённый endpoint проходит через `requirePermission(permission)`;
- frontend использует `permissionsService.can(permission)` и `*appCan` только для UI-видимости;
- при `mustChangePassword=true` пользователь блокируется от рабочих endpoint'ов до смены пароля.
- guest preview (`/guest-preview/:token`) работает как отдельный read-only контур: даже при наличии permission в payload backend запрещает любые write-методы через `guestReadonlyGuard`.
- управление пользователями (`users.manage`) включает CRUD-поток: создание, редактирование (имя/роль/статус), сброс пароля и удаление (кроме self-delete).
- защитное правило безопасности: пользователь с ролью `owner` не может изменить свой собственный `username` (логин), чтобы не терять устойчивую точку входа владельца;
- при удалении кастомной роли все пользователи этой роли автоматически переназначаются на системную роль `manager`.
- persistent login: при старте приложения выполняется попытка восстановления сессии через refresh flow (`initSession/initAuth`); если refresh недействителен — токены очищаются и пользователь возвращается на login.
- `remember me` в форме входа управляет местом хранения токенов:
  - включено: `localStorage` (сессия переживает закрытие вкладки/браузера),
  - выключено: `sessionStorage` (сессия живет до закрытия вкладки).

## Расчёт итогов КП

Только на фронтенде (`KpBuilderComponent` computed signals). В MongoDB **не хранятся**.

```
effectiveUnitPrice = round(item.price × (1 + markup%/100) × (1 - discount%/100))
  where:
  - markup% применяется только если `markupEnabled = true` (0..500)
  - discount% применяется только если `discountEnabled = true` (0..100)

subtotal  = Σ (effectiveUnitPrice × item.qty)
vatAmount = round(subtotal × vatPercent / (100 + vatPercent))
total     = subtotal
```

`subtotal` в КП трактуется как сумма **с НДС**; строка «В том числе НДС» выделяет налог из итоговой суммы.

UI-отображение в таблице КП:
- колонка цены: `Цена (вкл. НДС)`;
- колонка суммы: `Сумма (вкл. НДС)`;
- отдельная колонка: `НДС (в т.ч.)` по строке (`round(lineSum * vatPercent / (100 + vatPercent))`).

## Товары в КП — снимок данных

При добавлении товара данные **копируются** из каталога (`productId`, `code`, `name`, `description`, `unit`, `price`, `imageUrl`).  
Изменение товара в каталоге **не влияет** на существующие КП.

Дополнительно для загрузки фото в карточку товара:
- backend endpoint `POST /api/products/upload-image` автоматически удаляет прозрачные поля (alpha-trim) и может trim-ить белые поля для non-alpha изображений, если фон по краям near-white;
- геометрия контента сохраняется без растягивания/искажения;
- если trim не улучшает плотность кадра (размер не уменьшается), исходник остаётся без изменений;
- для уже загруженного архива локальных фото доступен one-time script: `npm run media:trim-products` (backend).

## Технический профиль товара (ProductSpec)

- Технические характеристики товара хранятся отдельно от `Product` в коллекции `ProductSpec` (один профиль на один товар, связь через `productId`).
- `Product` остаётся лёгкой сущностью каталога; загрузка/spec-edit выполняются только по требованию через отдельный API.
- Структура характеристик динамическая:
  - `groups[]` — разделы паспорта (`title`);
  - `params[]` внутри раздела — пары `name/value`.
- Поддерживаются отдельные чертежи `drawings.viewFront/viewSide/viewTop/view3D`.
- Если профиль отсутствует, товар продолжает работать в каталоге и КП без ошибок; техпаспорт создаётся по upsert-flow.
- Генерация PDF-паспорта допускается только при наличии `ProductSpec`; иначе API возвращает `404`.
- Для каталога товаров поддерживается фильтр `hasSpec` (`true/false`) на уровне API, чтобы быстро выделять заполненные/незаполненные позиции.
- В редакторе техпрофиля поддержан сценарий «Копировать из аналога»: характеристики можно перенести из другого товара серии, с опцией копирования/пропуска блока чертежей.

## Логика редактора КП

- Добавить товар: если уже есть — `qty++`, иначе новая позиция с `qty=1`
- В `KP Builder` доступен быстрый сценарий `+ Добавить товар`: открывается compact-форма в табличном стиле с минимальными полями строки КП (`Артикул`, `Наименование`, `Описание`, `Ед.`, `Цена`), чтобы ввод был как в таблице документа.
- Для корректировок цены используется массовое применение к выбранным строкам:
  - у позиции есть чекбокс `Выбрать`;
  - в панели `Состав КП` задаются `% Наценка` (до 500) и `% Скидка` (до 100);
  - кнопки `Применить к выбранным` записывают значения в поля позиции;
  - кнопка `Сбросить наценку/скидку` очищает корректировки у выбранных позиций.
- В `Состав КП` всегда показывается фото позиции (если есть `imageUrl`)
- Изменить qty: минимум 1
- Удалить позицию: явной кнопкой `Удалить` для конкретной строки (по `productId`)
- Автосохранение: debounce 2 сек после изменения в КП, но только после первой добавленной товарной позиции (`items.length > 0`) → `PUT /api/kp/:id`
- Ручное сохранение: немедленно, сбрасывает debounce
- Локальная отказоустойчивость: параллельно с autosave хранится browser-backup (`localStorage`, ключ `kp_builder_backup_<kpId>`); после reload пользователь получает модалку восстановления.
- Undo/Redo: редактор хранит до 10 последних снапшотов состояния КП; hotkeys `Ctrl+Z`, `Ctrl+Y`, `Ctrl+Shift+Z`.
- Блок `Параметры КП` в `KpBuilder` редактирует `metadata.number`, `metadata.validityDays`, `metadata.prepaymentPercent`, `metadata.productionDays`, `vatPercent`
- В `Параметры КП` доступно переключение `kpType` и выбор шаблона брендирования для текущего документа (`PUT /api/kp/:id/switch-type`); товарные позиции сохраняются, а зависимые snapshot-поля пересобираются сервером.
- Для `switch-type` действует optimistic flow: UI обновляется мгновенно; при backend-ошибке состояние откатывается к последнему валидному snapshot и показывается уведомление.
- Поле `metadata.tablePageBreakAfter` задаёт, после какой строки таблицы делать перенос на новую страницу (используется как `itemsPerPage` в документе)
- `metadata.defaultMarkupPercent/defaultDiscountPercent` — source defaults для bulk-полей наценки/скидки в редакторе; значения приходят из карточки нашей компании и сохраняются в КП.
- Для нового КП значения по умолчанию берутся из `Settings` на бэкенде (frontend не должен подставлять локальные дефолты)
- Резервные копии (MongoDB + media) управляются централизованно через `/settings` (раздел "Бэкапы"): ручной запуск, поиск/фильтрация, скачивание, удаление и очистка старше N дней; операции требуют `backups.manage`
- Разрушительные действия в UI (`удаление записи справочника`, `удаление/cleanup бэкапов`) подтверждаются только через `ModalService` + `ui-modal`; системный `window.confirm` не допускается.
- Новый получатель из редактора КП: кнопка «+» открывает ту же форму контрагента в модальном окне на странице КП (`shared/components/counterparty-form`); после сохранения контрагент попадает в локальный список сайдбара и сразу выбирается как получатель (`Kp.recipient` — snapshot). Уход со страницы при несохранённом КП — через `ui-modal`, не системный `confirm`
- Snapshot получателя в KP Builder отображается как readonly-блок с пояснением; замена получателя разрешена только в статусе `draft` и применяется сразу при выборе в `select` (без дополнительного confirm-диалога).
- Смена статуса КП из toolbar подтверждается модальным диалогом; в UI подсказки переходов завязаны на разрешённые status transitions.
- При создании КП на Home пользователь нажимает только `+ Создать КП`; `companyId` и `kpType` могут не передаваться: backend берёт компанию-инициатора по default-логике (`isDefaultInitiator`, иначе первая активная наша компания) и `kpType=standard`.
- Выбор компании/типа/шаблона выполняется в `KpBuilder` (блок `Параметры КП`); переключение применяется сразу при выборе, без отдельной кнопки.
- Для компании используются `brandingTemplates`: при `Авто` сервер сначала выбирает `isDefault=true` для заданного `kpType`, а если default не задан — берёт первый доступный шаблон этого `kpType`; если передан `templateKey`, используется ручной override (с проверкой `template.kpType`).
- Колонка `Описание` в документной таблице КП показывается только если хотя бы у одной позиции заполнено описание; если описания пустые у всех строк, колонка скрывается автоматически.
- Управление `brandingTemplates` вынесено из общей формы контрагента в отдельный менеджер шаблонов для наших компаний (отдельная модалка из списка `Counterparties`), чтобы не смешивать реквизиты компании и брендирование КП.
- Для шаблона типа `response` (Ответ на письмо) в менеджере показываются только поля, релевантные письму: обязательный `assets.kpPage1`; дополнительные фоны (`kpPage2/passport/appendix`) скрываются и очищаются.
- Создание КП без валидного шаблона запрещено: как минимум обязателен `assets.kpPage1`; при ошибке возвращается `400` с явным текстом.
- В `Kp.companySnapshot` сохраняется расширенный immutable snapshot (`templateKey/templateName/kpType/assets/texts/requisitesSnapshot`); дальнейшие правки карточки компании не изменяют уже созданное КП.
- При смене типа КП условия (`conditions`) заменяются на шаблонные только если текущий список всё ещё совпадает с предыдущим шаблоном (или явно передан флаг `overwriteConditions=true`); вручную изменённые условия сохраняются.

## Многостраничность документа A4

`KpDocumentComponent` разбивает items на страницы (`itemsPerPage = 6` по умолчанию):
- Стр. 1: фон из `companySnapshot.assets.kpPage1` + `KpHeaderComponent` (получатель + метаданные)
- Стр. 2+: фон из `companySnapshot.assets.kpPage2` (если не задан — без фона), без шапки, `margin-top: 20mm`
- Последняя стр.: `KpTableComponent` (итоги + условия) + `companySnapshot.texts.footerText` (HTML)
- Текстовые блоки шаблона: `headerNote` (под шапкой), `introText` (перед таблицей), `closingText` (после таблицы)
- Smart Variables в текстах поддерживаются в формате `{{token}}` (например `{{client_name}}`, `{{kp_number}}`, `{{total_amount}}`, `{{date}}`, `{{manager_name}}`).
- Подстановка токенов централизована в `KpTemplateService` (source-of-truth = `KpBuilderStore`), а в шаблонах документа применяется через pipe `kpTemplate`.
- Неизвестные токены не удаляются и остаются как есть, чтобы избежать «тихой» потери текста при ошибках шаблона.
- Номера страниц — если страниц > 1

## Печать / PDF

`window.print()` — браузерная печать.  
Элементы с классом `.no-print` скрываются через `@media print`.  
Скрывается: тулбар builder, сайдбар, кнопки действий.  
Остаётся: только `KpDocumentComponent` (страницы A4).

HQ-экспорт (`GET /api/kp/:id/export`) генерируется на backend через Puppeteer:
- включён `displayHeaderFooter`;
- header содержит название КП и дату;
- footer содержит нумерацию `Страница X из Y`.

Для многостраничных документов применяются utility-классы переноса:
- `.page-break-before` — принудительный перенос на новый лист;
- `.avoid-break` — запрет разрыва блока/строки внутри страницы.

## Контрагенты

- Одна компания может совмещать роли (`role: ['client', 'supplier']`)
- Отдельная роль `company` используется для нашей компании (`isOurCompany=true`) как источника реквизитов/фона/footer в КП
- При сохранении контрагента роль `company` синхронизируется с флагом `isOurCompany`, чтобы список компаний-инициаторов на Home и карточка контрагента не расходились.
- Для обратной совместимости исторические карточки с ролью `company` считаются валидной компанией-инициатором даже если `isOurCompany` еще не проставлен.
- Для карточек нашей компании доступен раздел шаблонов брендирования (`brandingTemplates`): несколько шаблонов по `kpType`, с одним `isDefault` на тип.
- Для компаний-инициаторов КП действует отдельный флаг `isDefaultInitiator` (одна компания по умолчанию). При установке нового default флаг снимается с предыдущей компании автоматически.
- В каждом шаблоне редактируется список `conditions[]` (условия КП). Эти условия используются как дефолтные при создании нового КП для выбранного шаблона/типа.
- Для ранее созданных шаблонов (до перехода с `texts` на `conditions`) при открытии формы компании пустой `conditions[]` автоматически заполняется базовыми 3 условиями, чтобы редактор не показывал пустой блок.
- ИНН: 10 или 12 цифр для `ООО/ИП/АО/ПАО/МКУ/Другое`; для `Физлицо` поле не обязательно (если заполнено — проходит ту же цифровую валидацию)
- КПП: 9 цифр, только для юрлиц — валидация в Mongoose
- Поиск по ИНН: `GET /api/counterparties/lookup?inn=` → DaData API (требует `DADATA_TOKEN`)
- Для формы создания/редактирования: если `shortName` не заполнен, frontend отправляет fallback `shortName = name`, чтобы не конфликтовать с backend-required правилом.
- Для `Физлицо` форма работает в упрощенном режиме: в базовом блоке остаются ключевые поля (`форма`, `роль`, `ФИО/краткое имя`, `статус`, `телефон`, `email`), `ИНН` отображается как опциональный, остальные реквизиты доступны как дополнительные поля.
- Для `Физлицо` роль `company` скрывается в UI, чтобы снизить риск ошибочного создания карточки «нашей компании».

## Нумерация КП

Серверная генерация номера в `POST /api/kp`, `POST /api/kp/:id/duplicate` и `PUT /api/kp/:id/switch-type` (для авто-номеров).

Формат зависит от типа:
- `response`: `ПИСЬМО-001`, `ПИСЬМО-002`, ...
- остальные типы: `КП-001`, `КП-002`, ...

Правило:
- берутся только номера соответствующего type-prefix (`КП-\d+` или `ПИСЬМО-\d+`);
- находится максимальный существующий serial;
- новый номер = `max + 1` (с `padStart(3, '0')`).
