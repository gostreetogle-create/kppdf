## ОБЯЗАТЕЛЬНОЕ ПРАВИЛО — ОБНОВЛЕНИЕ ПАСПОРТА ПРОЕКТА

> Это правило применяется к КАЖДОМУ ответу, который изменяет код, схему БД, API или функциональность.
> Не только к spec-задачам. К любому чат-ответу.

### Алгоритм — выполнять в конце КАЖДОГО ответа

**Шаг 1.** Определить что изменилось в этом ответе:
- Новая/изменённая модель БД → обновить раздел 3
- Новый/изменённый эндпоинт → обновить раздел 4
- Новая/изменённая фича → обновить раздел 5
- Новый маршрут, компонент, изменение состояния → обновить раздел 6
- Закрытая задача из TODO → обновить раздел 8

**Шаг 2.** Обновить все затронутые разделы PROJECT_PASSPORT.md (не только историю!)

**Шаг 3.** Добавить запись в раздел 10 ИСТОРИЯ ИЗМЕНЕНИЙ:
```
| YYYY-MM-DD | Что конкретно изменилось |
```

### Что считается значимым изменением

- Новая фича или изменение существующей
- Новый или изменённый эндпоинт API
- Изменение схемы БД (поле, модель, индекс)
- Новый компонент UI kit
- Рефакторинг бизнес-логики
- Изменение деплоя или инфраструктуры
- Исправление бага из раздела 8

### Типичные ошибки которые нельзя допускать

❌ Добавить только запись в историю, не обновив разделы 3/4/5/6  
❌ Пропустить обновление потому что "задача небольшая"  
❌ Обновить паспорт частично — только один раздел из нескольких затронутых  
❌ Написать в истории "обновлено" без конкретики что именно изменилось  

### Проверочный вопрос перед завершением ответа

Спроси себя: *"Если другой ИИ прочитает только PROJECT_PASSPORT.md — он поймёт что я только что сделал?"*  
Если нет — паспорт не обновлён достаточно.
